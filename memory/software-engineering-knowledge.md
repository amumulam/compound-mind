## 🏗️ 架构设计原则

### 1. SOLID 原则

**S - 单一职责原则 (SRP)**
- 一个类/模块只负责一件事
- 变更原因只有一个
- 例子：UserService 只处理用户逻辑，不处理邮件发送

**O - 开闭原则 (OCP)**
- 对扩展开放，对修改关闭
- 通过抽象和接口实现
- 新功能通过扩展，不修改现有代码

**L - 里氏替换原则 (LSP)**
- 子类可以替换父类
- 不破坏继承体系
- 子类不应该改变父类的行为约定

**I - 接口隔离原则 (ISP)**
- 接口要小而专一
- 客户端不应该依赖不需要的接口
- 拆分胖接口

**D - 依赖倒置原则 (DIP)**
- 依赖抽象，不依赖具体实现
- 高层模块不依赖低层模块
- 通过接口/抽象类解耦

### 2. 设计模式分类

**创建型模式**
- 工厂模式 - 创建对象的接口
- 单例模式 - 全局唯一实例
- 建造者模式 - 复杂对象分步构建
- 原型模式 - 克隆创建对象

**结构型模式**
- 适配器模式 - 接口转换
- 装饰器模式 - 动态添加功能
- 代理模式 - 控制对象访问
- 组合模式 - 树形结构

**行为型模式**
- 策略模式 - 算法可互换
- 观察者模式 - 事件通知
- 命令模式 - 请求封装
- 责任链模式 - 处理链

### 3. 架构风格

**分层架构**
```
┌─────────────────┐
│   Presentation  │  展示层
├─────────────────┤
│   Business      │  业务层
├─────────────────┤
│   Data Access   │  数据访问层
├─────────────────┤
│   Database      │  数据层
└─────────────────┘
```

**微服务架构**
- 服务独立部署
- 服务间通过 API 通信
- 每个服务有自己的数据库
- 适合大型复杂系统

**事件驱动架构**
- 生产者发布事件
- 消费者订阅事件
- 解耦、异步处理
- 适合高并发场景

## 💻 代码质量

### 1. 代码整洁之道

**命名规范**
- 有意义的名字：`getUserById` > `get`
- 避免缩写：`calculateTotal` > `calcTot`
- 布尔值用 `is/has/can` 前缀
- 常量全大写：`MAX_RETRY_COUNT`

**函数设计**
- 一个函数只做一件事
- 函数名描述做什么，不是怎么做
- 参数不超过 3 个，否则用对象
- 避免副作用

**注释原则**
- 代码即文档（自解释）
- 注释解释"为什么"，不是"是什么"
- TODO 注释要加上作者和日期
- 避免注释掉的代码

### 2. 代码审查清单

- [ ] 代码是否符合项目规范
- [ ] 是否有明显的 bug
- [ ] 是否有性能问题
- [ ] 是否有安全漏洞
- [ ] 是否有足够的测试
- [ ] 是否有清晰的文档
- [ ] 是否有重复代码
- [ ] 是否有过度设计

### 3. 重构技巧

**常见重构手法**
- 提取方法 - 减少重复
- 提取变量 - 提高可读性
- 内联方法 - 消除不必要的间接层
- 移动方法 - 放到正确的类
- 重命名 - 提高可读性

**重构时机**
- 添加新功能前
- 修复 bug 时
- 代码审查时
- 性能优化时

## 🧪 测试策略

### 1. 测试金字塔

```
        ╱╲
       ╱  ╲        E2E 测试（少量）
      ╱────╲       
     ╱      ╲      集成测试（中等）
    ╱────────╲     
   ╱          ╲    单元测试（大量）
  ╱────────────╲   
```

### 2. 单元测试原则

**AAA 模式**
```typescript
// Arrange - 准备
const user = { name: 'test' };

// Act - 执行
const result = userService.validate(user);

// Assert - 断言
expect(result).toBe(true);
```

**FIRST 原则**
- Fast - 快速执行
- Independent - 相互独立
- Repeatable - 可重复
- Self-validating - 自我验证
- Timely - 及时编写

### 3. 测试覆盖率

- 语句覆盖率 - 每行代码执行
- 分支覆盖率 - 每个 if/else 分支
- 函数覆盖率 - 每个函数调用
- 目标：核心逻辑 80%+，工具函数 60%+

## 🔧 开发实践

### 1. Git 工作流

**Git Flow**
```

           ╱      ╱
release ──●──────●
         ╱
develop ─●──────●──────●
         ╱      ╱

**Commit 规范**
```
feat: 添加新功能

### 2. 代码审查流程

1. 作者提交 PR/MR
2. CI 自动检查通过
3. 至少一人审查通过
4. 解决所有评论
5. Squash 合并
6. 删除分支

### 3. 持续集成/部署

**CI 流程**
```
代码提交 → Lint → Build → Test → 报告
```

**CD 流程**
```
测试通过 → 构建镜像 → 部署测试环境 → 自动测试 → 部署生产
```

## 🐾 爪爪的经验心得

### 务实原则

1. **不要过度设计** - YAGNI (You Aren't Gonna Need It)
2. **先让它工作，再让它正确，最后让它快** - Kent Beck
3. **简单设计 > 复杂设计** - KISS
4. **代码是给人看的，顺便给机器执行** - Donald Knuth

### 沟通原则

1. **技术决策要解释原因** - 为什么比是什么重要
2. **文档要更新** - 过期文档比没有文档更糟糕
3. **主动汇报** - 不要等主人问

### 学习方法

1. **官方文档优先** - 最权威的来源
2. **源代码次之** - 实际实现
3. **社区资源补充** - 经验分享
4. **实践验证** - 动手试试
