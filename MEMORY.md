## 学到的经验

### 2026-03-07

**严重错误：违反目录规范，文件放错位置**

**错误行为**：
- 在根目录创建 `plans/` 目录，而不是使用 `docs/plans/`
- 违反了 Compound Mind 框架的目录结构设计

**错误原因**：
- 没有在行动前查阅 AGENTS.md 的目录规范
- 违反了行事原则的第一条"先深度思考"
- 规范存在但没有被遵守

**正确做法**：
1. 行动前先查阅 AGENTS.md
2. 确认正确的目录位置
3. 然后再创建文件

**目录规范**：
| 内容 | 正确位置 |
|------|----------|
| 计划文档 | docs/plans/ |
| 解决方案 | docs/solutions/ |
| 头脑风暴 | docs/brainstorms/ |
| 每日日志 | memory/YYYY-MM-DD.md |

---

**约束力层级理论**

**核心认知**：
| 层级 | 机制 | 强度 | 适用场景 |
|------|------|------|----------|
| 声明式 | 文档定义规范 | 最弱 | 传播知识 |
| 检查式 | Git Hook / Heartbeat | 中等 | 防御检查 |
| 强制式 | 工具层面拦截 | 最强 | 核心约束 |

**框架局限性**：
- 框架只能"声明"和"检查"，无法"强制执行"
- 真正的强制执行需要 OpenClaw 核心支持
- Git Hook 虽然自动化，但有局限（不使用 git 就不触发）

**多层防护机制**：
- 事前：Session 启动时强制读取规范
- 事中：Heartbeat 定期扫描检查
- 事后：Git Hook 提交前检查
- 即使某一层失效，其他层仍可发现问题

### 2026-03-06

**严重错误：将维护者职责强加给用户**

**错误行为**：
- 在框架模板中添加观测日报监测
- 让所有框架用户都需要"监测框架是否正常"

**错误原因**：
- 监测是维护者的事，不是用户的事
- 增加用户负担和理解成本
- 框架如果需要监测才能用，说明不靠谱、不专业

**正确理解**：
| 角色 | 需求 |
|------|------|
| 框架用户 | 只需要核心功能 |
| 框架维护者 | 监测框架运行 |

**教训**：设计功能时要思考"这是谁的事？"
