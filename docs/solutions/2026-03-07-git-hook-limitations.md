# [[2026-03-07-git-hook-limitations]] - Git Hook 的致命局限

## 问题描述

在实现目录规范强制执行机制时，考虑使用 Git Hook 进行提交前检查，但发现 Git Hook 有严重的局限性。

---

## 现象

### 观察到的情况
- Git Hook 在 commit 前自动执行检查
- 如果用户不使用 git（只是创建文件但不提交），Hook 永远不会触发
- 检查机制完全依赖用户的 git 操作流程

### 期望结果
无论用户是否使用 git，目录规范都能被检查和执行

---

## 根因分析

### 直接原因
- Git Hook 是 git 生态系统的一部分，只对 git 操作生效
- 不使用 git 就不会触发检查

### 深层原因
- 任何基于 git 的检查机制都有同样的局限性
- 需要理解工具的本质能力边界

---

## 解决方案

### 方案描述
采用多层防护机制，不依赖单一的 Git Hook

### 实施步骤
1. **事前预防**：Session 启动时强制读取目录规范
2. **事中检查**：Heartbeat 定期扫描目录结构（独立于 git）
3. **事后拦截**：Git Hook 提交前检查（作为额外保护）

### 权衡
- Git Hook 仍然有用，但不能作为唯一防线
- 需要配合其他机制实现完整防护

---

## 验证方法

1. 即使不使用 git，Heartbeat 也能发现目录问题
2. 使用 git 时，Git Hook 提供额外的保护
3. 测试多层防护的整体效果

---

## 学到的经验

### 技术层面
- **理解工具本质**：Git Hook 只对 git 操作生效，不使用 git 就不触发
- **单一防护的脆弱性**：依赖单一机制容易被绕过
- **多层防护的价值**：即使某一层失效，其他层仍可发现问题

### 流程层面
- **约束力层级理论**：
  | 层级 | 机制 | 强度 | 适用场景 |
  |------|------|------|----------|
  | 声明式 | 文档定义规范 | 最弱 | 传播知识 |
  | 检查式 | Git Hook / Heartbeat | 中等 | 防御检查 |
  | 强制式 | 工具层面拦截 | 最强 | 核心约束 |
- **框架能力边界**：框架只能"声明"和"检查"，无法"强制执行"

---

## 预防措施

1. 设计检查机制时，考虑工具的本质能力边界
2. 采用多层防护，不要依赖单一机制
3. 清晰理解每种机制的适用场景和局限
