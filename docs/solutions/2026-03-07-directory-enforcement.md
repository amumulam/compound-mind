# [[2026-03-07-directory-enforcement]] - 目录规范强制执行机制

## 问题描述

框架定义了目录规范（计划文档应放在 `docs/plans/`，不是根目录的 `plans/`），但没有机制确保执行，导致文件放错位置。

---

## 现象

### 观察到的情况
- 在根目录创建了 `plans/` 目录
- 违反了 Compound Mind 框架的目录结构设计
- AGENTS.md 虽然定义了规范，但没有被遵守

### 期望结果
文件自动被放置在正确的目录位置，或者至少在放错位置时能被及时发现并纠正

---

## 根因分析

### 直接原因
- 没有机制确保规范被遵守
- 规范存在但没有执行力

### 深层原因
- 框架只能"声明"和"检查"，无法"强制执行"
- 真正的强制执行需要 OpenClaw 核心支持
- 单一层面的防护容易被绕过或失效

---

## 解决方案

### 方案描述
采用多层防护机制，通过事前预防、事中检查、事后拦截三层防护确保目录规范被遵守

### 实施步骤
1. **事前预防**：在 AGENTS.md 中添加目录规范表格，标记正确的目录位置
2. **事中检查**：在 HEARTBEAT.md 中添加目录结构检测规则，定期扫描
3. **事后拦截**：在 install.sh/update.sh/uninstall.sh 中使用标记机制保护用户内容
4. **添加 COMPOUND_MIND_START/END 标记**：保护框架内容不被用户随意修改
5. **更新框架版本**：发布 v1.3.0

---

## 验证方法

1. Heartbeat 检测到错误的目录时发出警告
2. 用户修改框架内容时，标记保护机制生效
3. 测试脚本验证所有防护层正常工作

---

## 学到的经验

### 技术层面
- **多层防护优于单层防护**：即使某一层失效，其他层仍可发现问题
- **标记机制的价值**：COMPOUND_MIND_START/END 标记可以清晰界定框架内容和用户内容
- **Git Hook 的局限**：如果用户不使用 git，Git Hook 永远不会触发

### 流程层面
- **约束力层级理论**：
  | 层级 | 机制 | 强度 |
  |------|------|------|
  | 声明式 | 文档定义规范 | 最弱 |
  | 检查式 | Git Hook / Heartbeat | 中等 |
  | 强制式 | 工具层面拦截 | 最强 |
- **框架能力边界**：框架只能"声明"和"检查"，无法"强制执行"
- **维护者 vs 用户职责**：监测框架运行是维护者的事，不是用户的事

---

## 预防措施

1. 在行动前先查阅 AGENTS.md 的目录规范
2. 遵循"思考 → 规划 → 同意 → 执行"的行事原则
3. 使用多层防护机制，不要依赖单一防护层
4. 明确界定框架内容和用户内容，避免越界修改
